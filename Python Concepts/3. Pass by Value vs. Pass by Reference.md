# Pass by Value vs. Pass by Reference in Python

Understanding how variables are passed to functions is crucial in programming. However, Python does something a bit unique compared to older languages like C++ or Java.

To understand Python, we first need to quickly define the two traditional methods.

---

# 1. The Traditional Concepts

## Pass by Value

When a variable is passed by value, the function receives a **copy** of the variable's value.

Think of it like giving someone a **photocopy of a recipe**.

If they spill sauce on their copy, your original recipe is completely unharmed.

### Result

Changes made inside the function **do not affect** the original variable outside the function.

---

## Pass by Reference

When a variable is passed by reference, the function receives a **direct link (reference or memory address)** to the original variable.

Think of it like sharing a **Google Doc**.

If someone edits the document, everyone sees the changes instantly.

### Result

Changes made inside the function **do affect** the original variable outside the function.

---

# 2. The Python Way: "Pass by Object Reference"

Python does **not** strictly use **Pass by Value** or **Pass by Reference**.

Instead, it uses a mechanism officially called **Pass by Object Reference** (also known as **Pass by Assignment**).

### The Golden Rule

Everything in Python is an **object**, and every variable is simply a **name tag** pointing to an object in memory.

When you pass a variable to a function, Python passes the **reference to the object**, not the object itself.

What happens next depends entirely on whether the object is:

- **Mutable** (can be changed)
- **Immutable** (cannot be changed)

---

# A. Immutable Objects (Acts like "Pass by Value")

### Examples

- Integers (`int`)
- Floats (`float`)
- Strings (`str`)
- Booleans (`bool`)
- Tuples (`tuple`)

When you pass an immutable object into a function, it behaves exactly like **Pass by Value**.

Because the object cannot be modified, any attempt to change it simply creates a **new object**, and the local variable inside the function points to that new object.

The original variable outside the function still points to the original object.

### Example

```python
def try_to_change_number(num):
    print(f"Inside (start): {num}")

    # Integers are immutable.
    # This creates a NEW integer object 20
    # and points the local variable 'num' to it.
    num = 20

    print(f"Inside (end): {num}")


my_number = 10

try_to_change_number(my_number)

# Original variable remains unchanged
print(f"Outside: {my_number}")
```

### Output

```text
Inside (start): 10
Inside (end): 20
Outside: 10
```

### What Happened?

Initially:

```
my_number
     │
     ▼
    10
```

Function call:

```
num
 │
 ▼
10
```

Then this line executes:

```python
num = 20
```

Now:

```
num
 │
 ▼
20

my_number
     │
     ▼
    10
```

The original integer was **never modified**.

---

# B. Mutable Objects (Acts like "Pass by Reference")

### Examples

- Lists (`list`)
- Dictionaries (`dict`)
- Sets (`set`)
- Custom Objects

Mutable objects can be modified **in place**.

When such an object is passed into a function, both the outside variable and the parameter point to the **same object**.

Changing the object changes it for everyone pointing to it.

### Example

```python
def try_to_change_list(my_list):
    print(f"Inside (start): {my_list}")

    # Modify the existing object
    my_list.append(4)

    print(f"Inside (end): {my_list}")


original_list = [1, 2, 3]

try_to_change_list(original_list)

print(f"Outside: {original_list}")
```

### Output

```text
Inside (start): [1, 2, 3]
Inside (end): [1, 2, 3, 4]
Outside: [1, 2, 3, 4]
```

### What Happened?

Initially:

```
original_list
      │
      ▼
[1, 2, 3]
```

Function call:

```
original_list
      │
      ▼
[1, 2, 3]
      ▲
      │
   my_list
```

Executing:

```python
my_list.append(4)
```

modifies the existing list.

Now:

```
original_list
      │
      ▼
[1, 2, 3, 4]
      ▲
      │
   my_list
```

Both variables still point to the same object.

---

# The Exception: Reassigning a Mutable Object

Even with mutable objects, using the assignment operator (`=`) creates a new object instead of modifying the existing one.

The local variable simply starts pointing somewhere else.

### Example

```python
def reassign_list(my_list):
    # Creates an entirely new list
    my_list = [9, 9]

    print(f"Inside: {my_list}")


original = [1, 2]

reassign_list(original)

print(f"Outside: {original}")
```

### Output

```text
Inside: [9, 9]
Outside: [1, 2]
```

### What Happened?

Initially:

```
original
   │
   ▼
[1, 2]
```

Function parameter:

```
original
   │
   ▼
[1, 2]
   ▲
   │
my_list
```

Executing:

```python
my_list = [9, 9]
```

creates a brand-new list.

Now:

```
original
   │
   ▼
[1, 2]

my_list
   │
   ▼
[9, 9]
```

The original list was never modified.

---

# Summary for DSA Interviews

If you pass an **integer**, **string**, or **tuple** to a helper function (such as in recursion) and try to modify it, the original value **will not change** outside the function.

If you pass a **list**, **dictionary**, **set**, or another mutable object to a helper function and modify it (using operations like `append()`, `pop()`, `add()`, etc.), the changes **will persist** outside the function.

This behavior is why recursive algorithms such as **DFS**, **BFS**, **Backtracking**, and **Graph Traversal** often pass a single shared:

- `visited` set
- `result` list
- `path` list
- `memo` dictionary

through every recursive call instead of creating new ones each time.
